# OGC 2026 Grand Shipyard Puzzle 알고리즘 종합 분석 리포트

## 0. Executive Summary

현재 알고리즘(`myalgorithm.py`)은 **Backward JIT 스케줄링(초기해)과 ALNS + CP-SAT(수리 탐색)**을 결합한 메타휴리스틱으로, 해의 타당성(Feasibility)은 100% 보장하는 안정적인 구조입니다. 하지만 본질적으로 "질이 낮은 해"에서 점수가 크게 떨어지지 않는 병목을 안고 있습니다. 가장 큰 이유는 **'타이트한 2D+시간 공간 패킹 다양성 부족'**과 **'CP-SAT의 후보 위치 생성 한계'**입니다.

- **60초에서 300초로 좋아지는 문제**: 블록 수가 적당하거나 공간이 넓어 시간적 여유만 있으면 무작위(Random) 후보 생성이 우연히 정답을 찾을 수 있는 문제들(예: prob_1, 4, 36, 38, 40)입니다. 평균 약 60~80%의 극적인 총점 개선을 보입니다.
- **300초를 줘도 안 좋아지는 문제**: 구조적으로 공간이 극도로 부족하여 "정교한 테트리스"가 필요한 문제(예: prob_13, 23, 24, 25, 35)입니다. 무작위 코너 앵커링 후보들로는 좁은 틈을 뚫지 못해 탐색이 정체되며, 개선율이 10% 내외에 불과합니다.
- **결론적 병목**: 목적함수를 지배하는 것은 **Obj1(지연, Tardiness)**입니다. 공간이 부족하면 납기가 급한 블록조차 밀려나고, 후보 다양성 부재로 ALNS가 이를 뚫지 못하고 있습니다.

## 1. 알고리즘 구조 분석 (myalgorithm.py 기반)

현재 알고리즘의 핵심 철학과 한계는 다음과 같습니다.

1. **초기해(Initialization) 설계의 역설**: 
   - `reverse=True`를 통해 **납기일(Due date)이 늦은 블록부터 배치**합니다. 이는 `find_latest_slot`과 결합되어 후반부 시간을 먼저 채우는 JIT(Just-In-Time) 철학입니다.
   - **장점**: 성공할 경우 모든 블록이 납기일에 딱 맞춰 배치되므로 Tardiness가 0이 됩니다.
   - **단점(리스크)**: 공간이 빡빡한 인스턴스의 경우, 나중에 배치되는 "납기가 급한(Due date가 빠른) 블록"들이 들어갈 2D 공간이 이미 막혀버려, 결국 맨 뒤(Tardy slot)로 밀려나 막대한 지연(Obj1) 페널티를 받게 됩니다.

2. **ALNS + CP-SAT Repair의 장단점**:
   - **장점**: `destroy_workload`, `destroy_tardiness` 등 문제 도메인에 특화된 파괴 연산자가 매우 훌륭합니다. CP-SAT를 통해 엄밀한 충돌 회피(레이어, 크레인 방해)를 보장합니다.
   - **단점(병목)**: CP-SAT는 **주어진 후보(Candidates)** 내에서만 선택합니다. `generate_candidates` 함수를 보면 고작 15개의 후보를 만들고, 위치(x,y)는 4개의 코너 앵커링이나 랜덤에 의존합니다. 이로 인해 좁은 틈새를 비집고 들어가는 '창의적인' 배치가 아예 불가능합니다.

3. **시간이 늘어날 때 개선되는 이유와 안 되는 이유**:
   - **왜 좋아지는가?**: 300초가 주어지면 SA(Simulated Annealing) 루프가 더 많이 돌면서, '랜덤 위치 앵커링'이 우연히 딱 맞는 자리를 찾아낼 확률이 높아집니다.
   - **왜 안 좋아지는가?**: 공간 제약이 극도로 빡빡한 경우, 랜덤 앵커링으로는 유효한 자리를 아예 찾을 수 없습니다. 따라서 Repair 후보들이 모두 충돌하거나 기각되며 탐색이 제자리걸음을 합니다.

## 2. 결과 데이터 전체 분석 (60s vs 300s)

두 CSV 파일(`test_results_60s.csv`, `test_results_300s.csv`)의 데이터 분석 결과입니다.

| 지표 / 경향 | 분석 내용 |
| :--- | :--- |
| **전체 성능 향상** | 60초 대비 300초에서 전체 문제 평균 총점이 약 **42% 향상**되었습니다. |
| **총점을 지배하는 항목** | **Obj1(Tardiness)이 압도적**입니다. Obj1이 발생하는 문제(예: prob_26, 31, 38)의 총점은 수천만~수억 단위입니다. 반면 Obj1이 0으로 수렴한 문제(예: prob_2, 8, 16)의 총점은 수천~수만 대입니다. |
| **시간 증가의 효과 (Objective별)** | 300초 증가는 **Obj1을 줄이는 데 결정적 역할**을 합니다. 하지만 Obj1이 이미 잡힌 문제에서는 Obj2(불균형)와 Obj3(선호도)를 줄이는 데는 큰 효과가 없습니다(CP-SAT 목적함수 한계). |
| **분산 (Variance)** | 공간이 넓은 쉬운 문제들은 Run 간 점수 분산이 매우 작습니다. 하지만 난이도가 높은(Obj1이 남는) 문제들은 운에 따라 점수가 2배~5배까지 출렁입니다(예: prob_31, 38). |
| **데이터 이상치 및 신뢰성** | [강한 근거] **prob_16 Run 5의 300초 실행 시간이 64598.5초(약 18시간)로 측정되었습니다.** 이는 `multiprocessing.Pool`의 Deadlock(자원 반환 실패) 또는 `find_earliest_tardy_slot`의 무한 루프 버그로 추정됩니다. 60초 실험에서도 `prob_2` Run 11이 751초를 기록하는 등, **타임리밋 강제 종료가 완벽하게 동작하지 않는 치명적 리스크**가 있습니다. |

## 4. 예제 분류 체계 제안

40개 인스턴스를 단순 점수가 아닌 알고리즘 반응성에 따라 구조적으로 분류했습니다.

### Group A: 시간만 주면 폭발적으로 좋아지는 유형 (Time-Responsive)
- **Problem 목록**: prob_1, prob_4, prob_6, prob_7, prob_36, prob_38, prob_39, prob_40
- **근거**: 60초 $\rightarrow$ 300초 시 점수 개선율이 60~80%에 달합니다. 
- **개선 전략**: 이 그룹은 탐색 시간(Iteration)이 부족했을 뿐입니다. 초기해의 질을 조금만 높이거나, Fast Fallback 발동 조건을 늦추면 60초 내에서도 300초 급의 성능을 낼 수 있습니다.

### Group B: 좁은 공간 테트리스 형 - 시간 증가 무효 (Space-Bottlenecked)
- **Problem 목록**: prob_13, prob_23, prob_24, prob_25, prob_30, prob_35
- **근거**: 개선율이 10% 내외에 불과하며, 300초를 주어도 여전히 막대한 Obj1이 발생합니다.
- **개선 전략**: 위치 후보 생성(`generate_candidates`) 방식을 완전히 뜯어고쳐야 합니다. 단순 코너 앵커링이 아니라, NFP(No-Fit Polygon) 기반 촘촘한 빈 공간 좌표 추출 알고리즘이 필수입니다.

### Group C: Obj1 완벽 정복 후 최적화 튜닝 형 (Balance/Pref-Dominant)
- **Problem 목록**: prob_2, prob_3, prob_5, prob_8, prob_10, prob_16, prob_17
- **근거**: 이미 60초만으로도 Obj1(지연)이 0에 수렴하여 10만 점 이하의 낮은 점수를 유지합니다.
- **개선 전략**: 점수를 더 낮추기 위해서는 CP-SAT 수식 내에서 Obj2와 Obj3의 가중치를 정밀하게 튜닝해야 합니다.

### Group D: 거대 스케일 구조적 재앙 형 (Massive Tardiness)
- **Problem 목록**: prob_26, prob_27, prob_28, prob_31, prob_32, prob_33, prob_37
- **근거**: 블록 수가 너무 많아 초기해 생성부터 꼬이며, 300초 후에도 총점이 수천만~수억 단위입니다.
- **개선 전략**: `destroy_tardiness` 파괴 연산자 한 번에 제거하는 블록 수를 대폭 늘려(Large Neighborhood Search) 지역 최적해(Local Optima)를 강력하게 빠져나와야 합니다.

## 5. 점수가 나온 이유를 더 깊게 설명 (코드 구조 관점)

1. **초기 정렬 역배치(Reverse Due-Date)의 나비효과**:
   초기해에서 납기가 늦은 블록을 억지로 뒤쪽 시간에 끼워넣습니다. 하지만 좁은 베이에서는 먼저 배치된 큰 블록 하나가 장기간 자리를 차지해버리면, 이후 탐색하는 "납기가 당장 내일인 급한 블록"들이 들어갈 2D 공간이 증발합니다. 결국 급한 블록들이 베이 맨 뒤로 밀려나 Obj1(지연)이 천문학적으로 치솟습니다.

2. **Obj2 (Balance) Proxy 최적화의 함정**:
   `repair_cpsat`의 목적함수에 들어있는 Obj2는 `proxy_obj2_penalty = u_factors[bay_id] * bay_workloads[bay_id]`입니다. 이는 "베이 간의 워크로드 차이"를 최소화하는 진짜 Obj2가 아닙니다. 그저 "현재 워크로드가 많은 베이에 블록을 넣는 것을 꺼리게" 할 뿐입니다. 따라서 진정한 의미의 밸런싱이 이루어지지 않아 Obj1이 해결된 이후에도 Obj2 점수가 잘 안 떨어집니다.

3. **Fast Fallback의 재앙**:
   초기해 생성 중 20% 시간이 지나면 빈 공간을 찾는 대신 베이 맨 끝 시간에 겹치지 않게 블록을 이어 붙이는(`_empty_bay_entry`) Fast Fallback이 발동합니다. 이것이 작동하는 순간 해당 블록들은 어마어마한 지연(Tardiness)을 갖게 되며, 이를 15개짜리 CP-SAT Repair 후보만으로 복구하는 것은 불가능에 가깝습니다.

4. **Post Optimize Time의 한계**:
   탐색 마지막에 실행되는 `post_optimize_time` 함수는 블록의 시간(`entry`, `exit`)만 앞당깁니다. 즉, **베이 이동이나 x,y 위치 변경이 전혀 일어나지 않습니다.** 따라서 이 후처리 로직은 Obj1(지연)을 깎아내는 데만 기여할 뿐, Obj2와 Obj3에는 1%도 영향을 주지 못합니다.

## 6. 더 낮은 점수를 위해 할 수 있는 것 (우선순위 부여)

아래는 즉시 효과를 볼 수 있는 순서대로 정렬된 개선안입니다.

### [A. 가장 먼저 시도할 것: 구현 난이도 하, 효과 최상]
- **무엇을**: CP-SAT 후보 생성(`generate_candidates`) 위치 다양화.
- **이유/효과**: 좁은 틈에 블록을 넣지 못하는 병목을 뚫기 위함. 기존 4개 코너 앵커링을 주변 블록의 외곽선 기반 밀착 코너(Bottom-Left 휴리스틱 변형)로 10~20개 생성.
- **적중 Objective**: Obj1 (압도적 개선 기대). Group B 문제들 성능 폭발.

### [B. 다음으로 시도할 것: 초기해 정렬 로직 다변화]
- **무엇을**: `initialization()`에서 한 번은 납기 늦은 순(현재), 한 번은 납기 빠른 순(EDD), 한 번은 면적 큰 순서로 3가지 초기해를 빠르게 만들고 가장 점수가 좋은 것을 Initial State로 채택.
- **이유/효과**: 잘못된 초기 정렬로 인해 공간이 꼬여버린 Group D 문제들의 원천적 해소.
- **구현 난이도**: 하 (정렬 기준만 바꿔 함수 3번 호출).

### [C. 중급 난이도: ALNS Operator 파괴 규모(Degree) 동적 조절]
- **무엇을**: 정체기(Stagnation) 감지 시 `num_remove`를 현재 20%에서 40~50%까지 일시적으로 올리는 Adaptive Large Neighborhood Search 도입.
- **이유/효과**: 60초 vs 300초 비교 시 개선이 없는 문제들은 좁은 지역 최적해(Local Optima)에 갇힌 것입니다. 한 번에 왕창 부수고 다시 짓는 과정이 필요합니다.

### [D. 고난도지만 잠재효과 큰 것: 진정한 Obj2 최적화]
- **무엇을**: CP-SAT 모델 내에 각 베이별 워크로드 합을 나타내는 변수(Variable)를 선언하고, 그 최대-최소 차이 자체를 목적함수로 모델링.
- **이유/효과**: Obj1이 0으로 잡힌 Group C 예제들에서 궁극적인 점수 단축을 이끌어냄.

## 7. 내가 추가로 꼭 알아야 할 것

1. **치명적인 버그(Deadlock) 가능성**: [강한 근거]
   300초 실험 중 `prob_16`에서 발생한 64598초 현상은 `multiprocessing.Pool`의 고질적인 Windows 데드락입니다. 제출 환경에서 만약 이런 현상이 한 번이라도 발생하면 **0점 또는 실격 처리**될 수 있습니다. `timelimit` 강제 종료를 멀티프로세싱 내부가 아닌 외부 Main Process 레벨에서 감시하고 Kill하는 로직(Process Timeout)을 반드시 추가해야 합니다.
2. **Post Optimize Time의 루프 위험**: [중간 근거]
   `post_optimize_time` 내의 시간 당기기 `while curr_entry >= r_time:` 로직은 시간 단위가 클 때 엄청난 반복을 유발할 수 있습니다. 한 칸씩(1) 빼는 대신 이진 탐색(Binary Search)이나 점프(Jump) 로직을 적용해 속도를 높여야 합니다.
