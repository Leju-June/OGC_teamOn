최적화 그랜드 챌린지 2026 (OGC 2026) 조선소 퍼즐: 블록을 배치하고, 시간 내에 해결하라OGC 2026 조직위원회, 2026년 5월 31일 (문서 버전 1.1)초록 (Abstract)최적화 그랜드 챌린지(OGC) 2026은 조선업에서 도출된 공간적 block scheduling(일정 계획) 문제를 다룹니다. 조선소에서 선박의 각 구조 block은 bay라고 불리는 고정된 작업 공간 내부에서 만들어집니다. Block은 불규칙하고 층이 있는(layered) 형태를 가진 3차원 객체이므로, 겹치지 않게 bay에 배치하는 것 자체가 어려운 기하학적 문제입니다. 게다가 모든 block은 release date, processing time, due date를 가지고 있으며, bay 공간을 공유하기 때문에 주변의 다른 block들로 인해 대기하거나 필요한 것보다 더 오래 머물러야 할 수도 있습니다. 따라서 알고리즘은 각 block에 대해 어느 bay로 갈지, 어디에 어떻게 배치될지, 그리고 언제 들어가고 나갈지를 한꺼번에 결정해야 합니다. 본 문서는 공식적인 문제 정의, 참가 가이드, 그리고 평가 기준을 제공합니다.문서 업데이트: 본 문서는 오류를 수정하거나 모호한 부분을 명확히 하기 위해 개정될 수 있습니다. 최신 버전은 항상 대회 웹사이트에서 확인할 수 있으며, 타이틀 페이지의 버전 번호를 통해 현재 읽고 있는 개정판을 알 수 있습니다. 참가자는 웹사이트에서 업데이트를 확인할 책임이 있으며, 중요한 변경 사항 역시 해당 웹사이트에 공지될 것입니다.1. 문제 설명 (Problem Description)1.1 용어 (Terminology)Bay: 작업이 수행되는 조선소 내의 물리적으로 분할된 작업 공간입니다. 각 bay $j\in M$은 고정된 크기 $W_{j}\times H_{j}$를 가지며, 여기서 $M:={1,2,...,m}$은 bay들의 집합입니다.Block: 선박의 대형 구조적 구성 요소를 나타내는 생산 단위입니다. 각 block $i\in N$은 3D 구조를 형성하는 $K_{i}$개의 다각형 layer로 구성되며, 여기서 $N:={1,2,....,n}$은 block들의 집합입니다. Block은 회전할 수 있으며, 가능한 orientation(방향) 옵션의 수는 $O_{i}$로 주어집니다.Release date $(R_{i})$: block i의 처리를 시작할 수 있는 가장 빠른 시간입니다.Due date $(D_{i})$: block i의 원하는 완료 시간입니다.Processing time $(P_{i})$: block i를 처리하는 데 필요한 시간입니다.Entry time $(ENTRY_{i})$: block i가 bay에 배치되는 시간입니다.Exit time $(EXIT_{i})$: block i가 bay에서 나가는 시간입니다.Tardiness $(T_{i})$: due date를 초과한 지연 시간으로, $T_{i}=max(0,EXIT_{i}-D_{i})$와 같이 정의됩니다.Workload $(L_{i})$: block i의 처리 기간 동안 필요한 총 작업량입니다.Preference $(S_{ij})$: block i가 bay j에 할당되는 것에 대한 선호도 점수입니다.1.2 문제 설정 (Problem Setting)조선소에서 block은 여러 생산 단계를 거치며, 그 중 일부는 bay 내부에서 수행됩니다. 각 bay는 크레인, 노동력, 운송 경로와 같은 자원이 공유되는 고정된 작업 공간입니다. Block은 시간적(temporal) 및 공간적(spatial) 제약 조건을 모두 만족하면서 bay에 할당되고 처리되어야 합니다. 각 bay는 공간적으로 겹치지 않는 한 동시에 여러 block을 수용할 수 있습니다. 각 block은 고유한 release date, processing time, due date를 가집니다. 따라서, 할당 및 일정 계획 결정은 공간적 및 시간적 실현 가능성을 동시에 고려해야 합니다.1.3 Bays각 bay는 독립적인 작업 공간이며, bay들 사이에는 물리적 또는 운영상의 분리가 존재합니다. 일반적인 조선소는 여러 개의 bay를 가지고 있으며, 조선소가 m개의 bay로 구성되고 각 bay j는 $W_{j}\times H_{j}$의 크기를 가진다고 가정합니다. 주어진 block에 대해 다음 결정들이 동시에 이루어져야 합니다: 1. 어느 bay에 할당할지, 2. bay 내의 위치와 orientation, 3. 언제 block을 bay에 넣을지, 4. 언제 block을 bay에서 빼낼지. 각 bay는 독립적인 크레인을 가지고 있어 bay 내의 모든 작업은 개별적으로 수행될 수 있다고 가정합니다. 올해 대회와 관련된 operation에는 두 가지가 있습니다: ENTRY와 EXIT이며, 이는 각각 block을 bay에 넣는 것과 bay에서 빼내는 것을 의미합니다. 특정 날짜에 각 bay의 크레인 운전자는 그날 수행할 operation 목록을 받게 됩니다. 하루에 수행할 수 있는 operation 수에는 제한이 없으므로, 당일의 작업 전에 얼마든지 많은 횟수의 작업을 수행할 수 있습니다. 예를 들어, processing time이 3이고 5일 차에 bay에 배치되도록 예정된(5일 차에 ENTRY) block을 고려해 보겠습니다. 그러면 이 block이 EXIT할 준비가 되는 가장 빠른 날짜는 8일 차입니다. 즉, 이 block은 5일 차부터 7일 차까지 처리될 수 있습니다. 이 경우, 각 작업으로 인해 bay의 형태가 달라지므로 작업 순서 또한 중요하다는 점에 유의해야 합니다. 그럼에도 불구하고, 모든 EXIT 작업은 어떤 ENTRY 작업보다도 이전에 완료되어야 한다고 가정합니다. 동일한 유형의 작업들 사이에서는, 알고리즘이 수행 순서를 결정해야 합니다.1.4 Blocks우리에게는 처리해야할 block들이 주어집니다. 각 block은 3차원 구조를 형성하는 여러 다각형 layer들로 구성된다고 가정합니다. 각 layer를 정의하는 다각형이 반드시 볼록다각형(convex)일 필요는 없습니다. 각 block i는 K개의 layer로 구성되며, 모든 layer는 동일한 높이를 가진다고 가정합니다. "비어있는(empty)" layer는 존재할 수 없다는 점에 유의하십시오. 또한, 인접한 두 layer는 반드시 물리적으로 부착되어 있음이 보장되므로, "떠 있는(floating)" layer는 불가능합니다. Block이 배치될 때, block의 orientation 또한 결정되어야 합니다. 이번 대회에서는 각 block에 대해 가능한 orientation 옵션의 수($O_i$)가 미리 정해진 문제 인스턴스를 제공합니다. 각 block i는 또한 workload $L_{i}$와 bay에 대한 preference 점수 $S_{ij}$를 가집니다. Workload는 해당 block에 대해 수행되어야 하는 작업량을 나타내며, bay들 간에 총 workload의 분산(variance)이 적은 솔루션이 선호됩니다. Preference 점수 $S_{ij}$는 block i를 bay j에 할당하는 것이 얼마나 선호되는지를 나타냅니다.1.5 내려야 할 결정들 (Decisions to Make)각 block i에 대해 다음 결정들이 이루어져야 합니다.Bay j로의 할당Bay 내의 배치 위치Orientation 선택Entry time ENTRY와 exit time EXIT1.6 제약 조건 (Constraints)제약 조건은 크게 세 가지 그룹으로 나뉩니다. 첫 번째는 모든 블록이 베이(bay)에 할당되고 ENTRY/EXIT(반입/반출) 작업이 보장되어야 한다는 할당 제약 조건입니다. 두 번째는 시간 제약 조건으로, 블록 작업이 사용 가능해진 이후 최소 요구 가공 시간(processing time) 이상 동안 수행되도록 보장합니다. 세 번째 그룹은 베이 내 블록들의 공간적 실현 가능성(spatial feasibility)에 관한 것입니다. 크레인 작업은 당일 작업 시간 전에 수행될 수 있으므로, 반입 시간 $ENTRY_i$와 반출 시간 $EXIT_i$를 가진 블록 $i$는 $[ENTRY_i, EXIT_i)$ 기간 동안 할당된 베이를 차지하게 됩니다.1.6.1 할당 제약 조건 (Assignment Constraints)할당 제약 조건 (Assignment constraint): 모든 블록은 반드시 단 하나의 베이에만 할당되어야 하며, 할당되지 않은 블록이 존재해서는 안 됩니다.작업 제약 조건 (Operation constraint): 모든 블록은 정확히 하나의 ENTRY 작업과 하나의 EXIT 작업을 가져야 합니다. 블록이 한 번 베이에 배치되면 이동하거나 회전할 수 없습니다.1.6.2 시간 제약 조건 (Temporal Constraints)출시일 제약 조건 (Release date constraint): 블록은 자신의 출시일(release date) 이전에는 작업을 시작할 수 없습니다: $ENTRY_i \ge R_i$.가공 시간 제약 조건 (Processing time constraint): 블록 작업은 반출 전에 완료되어야 합니다: $EXIT_i - ENTRY_i \ge P_i$.참고사항: 블록은 요구되는 가공 시간이 지난 후에도 베이에 더 "남아" 있을 수 있습니다. 이는 주변의 다른 블록들로 인해 크레인이 접근하지 못해 반출이 방해받을 때 발생할 수 있습니다. 만약 반출 시간 $EXIT_i$가 마감일(due date) $D_i$를 초과하면 양(+)의 지연 시간(tardiness)이 발생합니다. 일반적으로 블록은 $D_i - R_i > P_i$를 만족하므로, 다른 블록들과의 관계를 고려하여 더 나은 ENTRY/EXIT 시간을 결정할 수 있는 약간의 "여유(slack) 시간"이 존재합니다.1.6.3 공간 제약 조건 (Spatial Constraints)방향 $o$, 위치 $(x,y)$에 배치된 블록 $i$의 $l$번째 레이어 다각형을 $P_{i,l}^{o,x,y}$라고 정의합니다. 동일한 날, 동일한 베이에 위치한 두 블록 $i_1$과 $i_2$의 다각형 내부 점들의 집합($int()$) 사이에 교집합이 존재한다면($int(P_{i_1,l}^{o_1,x_1,y_1}) \cap int(P_{i_2,l}^{o_2,x_2,y_2}) \neq \emptyset$), 두 블록 사이에 충돌(collision)이 발생한 것으로 봅니다. 반대로 교집합이 없다면 충돌이 없는 상태(collision-free)입니다. 두 블록이 다각형의 모서리(edge)를 공유하는 것은 충돌이 없는 것으로 허용됩니다.임의의 날짜 $t$와 베이 $j$에 동일하게 할당되어 머무르고 있는 블록들의 집합을 $N(t,j)$라고 할 때, 다음 제약 조건을 만족해야 합니다:베이 포함 제약 조건 (Bay containment constraint): 각 블록은 할당된 베이 영역 내에 완전히 포함되어야 합니다 (그림 7(b) 위반 예시 참고).레이어 충돌 없는 제약 조건 (Layer collision-free constraint): 동일한 베이, 동일한 날짜에 존재하는 서로 다른 두 블록은 같은 높이 레벨($l$)에 있는 레이어끼리 서로 충돌하지 않아야 합니다. 그림 7(a)처럼 위에서 보았을 때(top view)는 겹쳐 보일지라도, 높이별 레이어가 다르면 공간적으로 실현 가능합니다.크레인 작업 제약 조건 (Crane operation constraint): 베이에서 블록을 배치(ENTRY)하거나 제거(EXIT)할 때, 크레인이 간섭 없이 블록을 수직으로 이동할 수 있어야 합니다. 즉, 블록을 넣거나 빼려면 해당 블록의 레이어가 동일하거나 더 높은 레벨에 있는 다른 블록들의 레이어와 공간적 충돌이 없어야 합니다. 그림 7(d)의 경우, 블록 1이 이미 존재하는 상태에서 블록 2를 넣거나 빼려고 하면 아래쪽 레이어가 걸리므로 제약 조건을 위반하게 됩니다. 반면 블록 1은 블록 2가 아래에 있더라도 위가 뚫려 있으므로 크레인으로 들어 올릴 수 있습니다.1.7 목적 함수 (Objective Function)본 과제의 목적은 아래의 세 가지 기준을 가중치 조합한 함수를 최소화하는 것입니다:총 지연 시간 ($Z_1$): 블록들의 총 지연 시간의 합입니다.$$Z_1 := \sum_{i=1}^{n} T_i$$작업량 불균형 ($Z_2$): 베이 간의 최대 정규화된 작업량 불균형 값입니다.$$Z_2 := \max_{j_1, j_2 \in M, j_1 \neq j_2} \left| u_{j_1}\sum_{i \in N(j_1)} L_i - u_{j_2}\sum_{i \in N(j_2)} L_i \right|$$여기서 $L_i$는 블록 $i$의 작업량이며, 베이 가중치 $u_j$는 모든 베이의 평균 면적을 특정 베이 $j$의 면적으로 나눈 값입니다 ($u_j = \frac{\text{평균 면적}}{W_j \times H_j}$). 면적이 큰 베이는 가중치가 작게 부여되는데, 이는 큰 베이에 블록을 배치하는 것이 혼잡도를 줄여준다는 현실을 반영한 것입니다.총 선호도 점수 하락 ($Z_3$): 할당된 베이에 대한 선호도 점수 손실의 총합입니다.$$Z_3 := \sum_{j \in M}\sum_{i \in N(j)} (S_i^{\max} - S_{ij})$$여기서 $S_i^{\max}$는 블록 $i$가 가질 수 있는 가장 높은 베이 선호도 점수입니다. 모든 블록이 자신이 가장 선호하는 베이에 배정되면 $Z_3 = 0$이 됩니다.최종 적으로 최소화해야 하는 목표 함수는 다음과 같으며, $w_1, w_2, w_3$은 주어지는 음이 아닌 가중치 매개변수입니다.$$\text{Minimize } w_1 Z_1 + w_2 Z_2 + w_3 Z_3$$2 문제 인스턴스 정의 (Problem Instance Definition)참가자에게는 실제 조선소 문제를 모방하여 제작된 JSON 형식의 훈련용(training) 문제 인스턴스가 제공됩니다. 데이터는 다음과 같이 네 가지 파트로 구성됩니다:문제 이름 ("name")베이 목록 ("bays"): 각 베이의 너비(width)와 높이(height) 정보.블록 목록 ("blocks"): 각 블록의 release_time, due_date, processing_time, workload, 베이별 선호도 점수 리스트(bay_preferences), 그리고 방향별 레이어 다각형 좌표 리스트(shape).목적 함수 가중치 ("weights")인덱스 방식 참고: 수식(Section 1)에서는 1부터 시작하는 인덱스(1-based)를 사용했지만, 실제 JSON 데이터와 알고리즘 솔루션에서는 코딩의 편의(Pythonic)를 위해 0부터 시작하는 인덱스(0-based)를 사용합니다 (예: 베이 0 ~ $m-1$, 블록 0 ~ $n-1$).좌표계 및 기준점: 각 방향별 최하단 레이어(레이어 0)의 첫 번째 정점은 항상 [0.0, 0.0]이며 이를 기준점(reference point)이라 부릅니다. 알고리즘이 블록의 위치를 $(x,y)$로 결정하면, 이 기준점이 베이 원점으로부터 $(x,y)$만큼 평행 이동하게 됩니다. 정점 좌표는 소수점 아래 4자리까지 가질 수 있으며, 모양이 불규칙하므로 충돌 체크 시 수치적 불안정성이 생길 수 있습니다. 평가 서버와 동일한 검증을 위해 제공된 utils.py 내 함수를 활용하는 것이 강력히 권장됩니다.3 알고리즘 제출 및 평가 (Algorithm Submission and Evaluation)참가팀은 구현한 알고리즘 코드를 submission@optichallenge.com으로 이메일을 보내 제출해야 합니다. 제출된 코드는 "숨겨진(hidden)" 인스턴스를 사용해 전용 서버에서 평가 및 랭킹이 매겨지며 리더보드에 반영됩니다.3.1 알고리즘 제출 요구사항등록된 이메일 주소로 전송해야 하며, 하나의 zip 파일을 첨부해야 합니다.zip 파일의 최상위 경로(root)에 myalgorithm.py가 포함되어 있어야 합니다.만약 utils.py를 zip에 포함하더라도 이는 평가 시 원본 파일로 덮어씌워지므로 수정할 수 없습니다.zip 파일 크기는 15MB를 초과할 수 없습니다.한 번 제출이 수락되면 12시간의 재사용 대기 시간(cooldown period)이 적용되어 그동안은 새로 제출할 수 없습니다. 거부(reject)된 경우에는 즉시 재제출이 가능합니다.3.2 알고리즘 평가 서버 사양 및 제한사양: AMD Ryzen Threadripper PRO 9955WX, Ubuntu 24.04 LTS.시간 제한: 문제마다 다르며 구체적인 제한 시간은 대회가 끝난 후 공개됩니다 (보통 수 분에서 30분 사이). 시간 초과 시 알고리즘 크래시와 동일하게 실패 처리되므로 코드 내부에서 경과 시간을 철저히 모니터링해야 합니다.자원 및 환경 제약: * 실행 중 외부 인터넷 접속 불가실행 폴더의 상위 디렉터리 접근 불가최대 4개의 CPU 코어 (400% CPU 사용량 제한)최대 16GB 메모리제약 위반 시 실격될 수 있으며, 제어는 firejail과 cpulimit 도구로 강제됩니다.3.3 리더보드 점수 산정 방식각 테스트 인스턴스별로 다음과 같이 점수가 부여됩니다:솔루션이 실현 불가능(infeasible)하거나, 시간 초과, 혹은 크래시가 발생한 경우: -1점솔루션이 실현 가능(feasible)한 경우: R -
